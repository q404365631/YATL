# YATL (Yet Another Testing Language) - Руководство по использованию

YATL — это легковесный фреймворк для тестирования API с использованием YAML-спецификаций. Он поддерживает HTTP-запросы, валидацию ответов, извлечение данных и шаблонизацию с Jinja2.

## Структура теста

Файл теста YATL — это YAML-документ со следующими ключами верхнего уровня:

- `name` (опционально): описательное название набора тестов.
- `base_url` (опционально): базовый URL для всех запросов в тесте.
- `skip` (опционально): если установлено в `true`, весь тест будет пропущен во время выполнения.
- `steps`: список шагов теста, каждый из которых представляет HTTP-запрос и его проверки.

Каждый шаг может содержать следующие поля:
- `name` (опционально): название шага для идентификации в логах
- `description` или `desc` (опционально): подробное описание цели шага
- `skip` (опционально): если `true`, шаг будет пропущен
- `request` (обязательно): спецификация HTTP-запроса
- `expect` (опционально): проверки ответа

Пример:

```yaml
name: Тест API пользователей
base_url: http://localhost:8000

steps:
  - name: Создание пользователя
    description: Создание нового пользователя в системе с указанными данными
    request:
      method: POST
      url: /users
      body:
        json:
          id: 1
          name: Иван Иванов
          email: ivan@example.com
    expect:
      status: 200
```

Вы можете полностью пропустить тест, установив `skip: true` на верхнем уровне. Это полезно для временного отключения теста без его удаления.

Пример:

```yaml
name: Пропущенный тест
base_url: http://localhost:8000
skip: true

steps:
  - name: Этот шаг не будет выполнен
    request:
      method: GET
      url: /ping
```

Отдельные шаги также можно пропускать, добавив `skip: true` внутри шага. Такой шаг будет проигнорирован, а контекст не изменится.

Пример:

```yaml
steps:
  - name: Активный шаг
    request:
      method: GET
      url: /api/health
  - name: Пропущенный шаг
    skip: true
    request:
      method: POST
      url: /api/data
  - name: Следующий шаг
    request:
      method: GET
      url: /api/status
```

## Спецификация HTTP-запроса

Каждый шаг содержит объект `request` со следующими полями:

- `method` (обязательно): HTTP-метод (GET, POST, PUT, DELETE и т.д.). По умолчанию GET.
- `url` (обязательно): путь к эндпоинту, относительно `base_url`.
- `headers` (опционально): словарь HTTP-заголовков.
- `params` (опционально): query-параметры в виде пар ключ-значение.
- `cookies` (опционально): куки в виде пар ключ-значение.
- `body` (опционально): тело запроса, которое может быть JSON, XML, обычным текстом или form-данными.

### Типы тела запроса

YATL поддерживает несколько форматов тела:

#### JSON

```yaml
body:
  json:
    key: значение
    nested:
      field: 123
```

Заголовок `Content-Type` автоматически устанавливается в `application/json`, если не указан иначе.

#### XML

```yaml
body:
  xml: |
    <note>
      <to>Пользователь</to>
      <from>YATL</from>
    </note>
```

`Content-Type` будет установлен в `application/xml`.

#### Обычный текст

```yaml
body:
  text: "Привет, мир!"
```

`Content-Type` будет установлен в `text/plain`.

#### Form-данные

```yaml
body:
  form:
    username: ivan
    password: секрет
```

`Content-Type` будет установлен в `application/x-www-form-urlencoded`.

#### Multipart-файлы

```yaml
body:
  files:
    file: /путь/к/файлу.txt
```

### Таймаут

Вы можете указать таймаут запроса (в секундах):

```yaml
request:
  timeout: 30
```

## Валидация ответа

После отправки запроса вы можете проверить ответ с помощью блока `expect`.

### Код статуса

```yaml
expect:
  status: 200
```

### Заголовки

```yaml
expect:
  headers:
    content-type: application/json
    cache-control: no-cache
```

Валидация заголовков нечувствительна к регистру и игнорирует параметры (например, `charset`).

### Тело ответа

Вы можете валидировать JSON, XML или обычный текст.

#### Валидация JSON-тела

```yaml
expect:
  body:
    json:
      id: 1
      name: Иван Иванов
```

Поддерживаются вложенные объекты. Валидатор проверяет точное равенство.

Для проверки глубоко вложенных полей можно использовать **точечную нотацию** (как в блоке `extract`). Ключ с точкой интерпретируется как путь к значению в JSON-ответе.

Пример: если ответ содержит `{"user": {"profile": {"email": "test@example.com"}}}`, вы можете проверить email так:

```yaml
expect:
  body:
    json:
      "user.profile.email": "test@example.com"
```

Точечная нотация также работает с массивами (например, `items.0.name`).

#### Валидация XML-тела

```yaml
expect:
  body:
    xml:
      "/note/to": "Пользователь"
      "/note/from": "YATL"
```

Ключи — это XPath-выражения, значения — ожидаемое текстовое содержимое соответствующего элемента.

#### Валидация обычного текста

```yaml
expect:
  body:
    text: "Привет, мир!"
```

Валидатор проверяет, содержится ли ожидаемая подстрока в тексте ответа.

## Извлечение данных

Вы можете извлекать значения из ответа и сохранять их в контексте теста для использования в последующих шагах.

Используйте блок `extract`:

```yaml
extract:
  user_id: id
  token: access_token
```

Механизм извлечения зависит от типа содержимого ответа. YATL автоматически определяет формат ответа (JSON, XML или текст) на основе заголовка `Content-Type`, а также анализирует содержимое, если заголовок отсутствует.

### Извлечение из JSON

Для JSON-ответов вы можете указать путь (ключ) в JSON-объекте. Если путь равен `null`, в качестве пути используется имя ключа. Поддерживается точечная нотация для извлечения вложенных полей.

Пример: если ответ `{"id": 123, "user": {"email": "test@example.com"}}`, вы можете извлечь email следующим образом:

```yaml
extract:
  user_email: user.email
```

Также можно извлекать элементы массивов, используя индекс (например, `items.0.name`).

### Извлечение из XML

Для XML-ответов укажите XPath-выражения:

```yaml
extract:
  note_to: "/note/to"
```

Если XPath не указан (значение `null`), в качестве имени тега используется ключ.

### Извлечение из обычного текста

Для обычного текста можно использовать регулярные выражения:

```yaml
extract:
  match: "\d+"
```

Если шаблон не указан, извлечение не производится (требуется явное указание).

## Шаблонизация с Jinja2

YATL использует шаблонизацию Jinja2 для динамической генерации значений в запросах, ожиданиях и извлечениях. Контекст включает:

- Переменные, извлечённые из предыдущих шагов.
- `base_url` и название теста.
- Любые пользовательские переменные, определённые в тесте.

### Использование в YAML

Заключайте шаблонные строки в двойные фигурные скобки:

```yaml
request:
  url: /users/{{ user_id }}
  headers:
    Authorization: Bearer {{ token }}
```

### Пример

```yaml
steps:
  - name: Вход
    request:
      method: POST
      url: /login
      body:
        json:
          username: admin
          password: secret
    extract:
      token: access_token
  - name: Получить профиль
    request:
      method: GET
      url: /profile
      headers:
        Authorization: Bearer {{ token }}
```

## Запуск тестов

### Командная строка

Самый простой способ запустить тесты — использовать встроенный раннер:

```bash
python -m src.yatl.run
```

Эта команда рекурсивно ищет файлы с расширением `.yatl.yaml` или `.yatl.yml` в текущем каталоге и запускает их.

### Программное использование

Вы также можете интегрировать YATL в свои Python-скрипты:

```python
from yatl.run import Runner
from yatl.extractor import DataExtractor
from yatl.render import TemplateRenderer

runner = Runner(DataExtractor(), TemplateRenderer())
runner.run_test("path/to/yatl.yaml")
```

## Примеры

### Тест JSON API

```yaml
name: Пример JSON API
base_url: https://api.example.com

steps:
  - name: Создать элемент
    request:
      method: POST
      url: /items
      body:
        json:
          name: "Виджет"
          price: 9.99
    expect:
      status: 201
      body:
        json:
          success: true
    extract:
      item_id: id

  - name: Получить элемент
    request:
      method: GET
      url: /items/{{ item_id }}
    expect:
      status: 200
      body:
        json:
          name: "Виджет"
          price: 9.99
```

### Тест XML API

```yaml
name: Тест XML-сервиса
base_url: http://localhost:8000

steps:
  - name: Получить XML
    request:
      method: GET
      url: /xml
    expect:
      status: 200
      headers:
        content-type: application/xml
      body:
        xml:
          "/note/to": "Пользователь"
          "/note/from": "YATL"
```

### Тест обычного текста

```yaml
name: Текстовый эндпоинт
base_url: http://localhost:8000

steps:
  - name: Получить приветствие
    request:
      method: GET
      url: /text
    expect:
      status: 200
      body:
        text: "Привет"
```

## Расширенные возможности

### Условные шаги

В настоящее время YATL не поддерживает условные шаги нативно, но вы можете реализовать условную логику, написав отдельные файлы тестов.

### Логика повторных попыток

Не встроена; рассмотрите использование внешних инструментов, таких как `pytest` с декораторами повторов.

### Переменные окружения

Вы можете использовать переменные окружения в шаблонах через `os.environ` Jinja2:

```yaml
request:
  url: {{ os.environ.get("API_BASE_URL") }}/endpoint
```

### Пользовательские валидаторы

Расширьте класс `ResponseValidator`, чтобы добавить собственную логику валидации.

## Устранение неполадок

### Распространённые ошибки

- **Несоответствие заголовков**: убедитесь, что значения заголовков нормализованы (например, `content-type` может включать charset).
- **Ошибка извлечения JSON**: проверьте, существует ли указанный путь в ответе.
- **Ошибки шаблонизации**: проверьте, что переменные определены в контексте.

### Отладка

Включите подробное логирование, изменив `run.py` или используя уже присутствующие операторы `print`.
